## Wheel-Size API Rate Limit Fix

**Issue:** Got email from Wheel-Size saying "above 100% limit utilization" - 9,970 API hits in 24 hours (limit 100/hour).

**Root Cause Analysis:**
- `wheelSizeApi.ts` had `cache: "no-store"` on every request - no caching at all
- Legacy fallback in `fitment-search` route calls `importVehicleFitment()` for every vehicle NOT in DB
- Each import = 4+ API calls: getMakes() → getModels() → getModifications() → getVehicleData()
- Bots/crawlers or users browsing many vehicles triggered thousands of redundant calls

**Fix deployed:**
- Added in-memory caching to `wheelSizeApi.ts`
- Cache TTLs: Makes/Models/Years = 24h, Modifications = 12h, Vehicle data = 7 days
- Added cache utility functions (getCacheStats, pruneExpiredCache, clearCache)
- Commit: `ff81933` - "Add in-memory caching to Wheel-Size API to reduce API calls"

**Expected impact:** ~400 API calls/day max (vs 10,000) with normal usage patterns.

### Follow-up Fix: Models endpoint not using cache
**Issue:** Vehicle selector models dropdown not loading (empty) due to rate limiting.

**Root cause:** `/api/vehicles/models` was making DIRECT API calls to Wheel-Size, bypassing the cached `wheelSizeApi` module.

**Fix:** Updated models endpoint to use `getModels()` and `findMake()` from the cached wheelSizeApi module.
- Commit: `57ef15a` - "Fix: Use cached wheelSizeApi module in /api/vehicles/models"

**Notes:**

- Plan: Use WheelPros as primary when it's back online

### Additional Fallbacks Added

## Submodel/Trim Supplements for Older Vehicles

**Issue:** 2010 Chevy Avalanche (and other older vehicles) only showing "Base" trim instead of actual trims like LS, LT, LTZ. This happens when Wheel-Size API only returns engine codes (e.g., "5.3i") instead of trim names.

**Root Cause:**
- Wheel-Size API stores modifications by engine type, not trim level for older vehicles
- The "poor data" detection in trims endpoint wasn't catching labels like "Base (5.3i)"

**Fix deployed:**
1. Added submodel supplements for:
   - Chevrolet Avalanche (2002-2006: 1500, Z66, Z71, 2500; 2007-2013: LS, LT, LTZ)
   - Older Silverado/Silverado 1500 (1999-2006, 2007-2013 with proper trims)
   - Older Tahoe (2000-2006, 2007-2014)
   - Older Suburban (2000-2006, 2007-2014)

2. Fixed "poor data" detection in trims endpoint:
   - Now properly detects `"Base (5.3i)"` as poor data (was only checking exact `"base"`)
   - Added pattern to catch engine codes in parentheses

- Commit: `526e9a7` - "Add submodel supplements for Avalanche, older Silverado/Tahoe/Suburban + fix poor-data detection"

**Result:** Older GM trucks/SUVs now show proper trim options (LS, LT, LTZ, Z71, etc.) from static supplements.

## Hub Ring Calculation Bug Fix (~16:30)

**Issue:** Hub rings not being calculated/added in the wheel listing page "Quick Add" flow.

**Root Cause:**
- The `Wheel` type in `/wheels/page.tsx` did NOT include `centerbore`
- Wheel mapping from API response didn't extract `centerbore` from `it.properties.centerbore`
- `WheelsStyleCard` received `undefined` for `wheelCenterBore`, causing hub ring status to be "skipped"

**Fix:**
1. Added `centerbore?: string` to the `Wheel` type
2. Extract centerbore from API response in the wheel mapping
3. Added `centerbore` to `WheelProsItem.properties` type for TypeScript
4. Enhanced logging in `AddToCartButton` and `WheelsStyleCard` for debugging

**Commits:**
- `fe5f319` - fix(hub-rings): Add wheel centerbore to listing page data flow
- `38c026f` - fix: Add centerbore to WheelProsItem type to fix TS build error

**Hub Ring Logic (Verified Working):**
- Required: wheel bore > vehicle hub (by >0.5mm) → hub ring added
- Optional: wheel bore ≈ vehicle hub (within 0.5mm) → no ring needed
- Skipped: missing vehicle or wheel center bore data

## Abandoned Cart Tracking System (~18:30)

**Feature:** Comprehensive abandoned cart tracking and admin visibility for recovery and metrics.

### Cart States
- **active**: Has items, user is actively browsing
- **abandoned**: No activity for 1 hour with items in cart
- **recovered**: Order was completed from a previously tracked cart
- **expired**: Old carts beyond 30 days

### Schema (migration 0013_abandoned_carts.sql)
```sql
abandoned_carts table:
- cart_id (unique, for recovery links)
- session_id, customer info (name, email, phone)
- vehicle info (year, make, model, trim)
- items (JSONB), item_count
- subtotal, estimated_total
- status, recovered_order_id, recovered_at
- timestamps: created_at, updated_at, last_activity_at, abandoned_at
- source, user_agent, ip_address
```

### Tracking Flow
1. User adds item to cart → CartTracker syncs to server (2s debounce)
2. User visits checkout → Faster sync with customer info capture
3. 1 hour of inactivity → Cart marked "abandoned" (via processAbandonedCarts)
4. Order completed → CartRecoveryHandler marks cart "recovered", clears local cart

### Dashboard Widgets
- **Active Carts**: Currently shopping users
- **Abandoned**: Carts with items but no activity for 1h
- **Recovered**: Previously abandoned carts that converted
- **Recovery Rate**: recovered / (abandoned + recovered) percentage
- **Abandoned Value**: Total $ in abandoned carts
- **Recovered Revenue**: Total $ recovered from abandoned carts
- **Recent Abandoned**: List of latest abandoned carts

### Abandonment Rules
- Minimum $10 cart value to track (skip empty/tiny carts)
- 1 hour inactivity → abandoned
- 30 days → expired
- Recovery: same cartId + order completion
