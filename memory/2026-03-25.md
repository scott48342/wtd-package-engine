## Wheel-Size API Rate Limit Fix

**Issue:** Got email from Wheel-Size saying "above 100% limit utilization" - 9,970 API hits in 24 hours (limit 100/hour).

**Root Cause Analysis:**
- `wheelSizeApi.ts` had `cache: "no-store"` on every request - no caching at all
- Legacy fallback in `fitment-search` route calls `importVehicleFitment()` for every vehicle NOT in DB
- Each import = 4+ API calls: getMakes() → getModels() → getModifications() → getVehicleData()
- Bots/crawlers or users browsing many vehicles triggered thousands of redundant calls

**Fix deployed:**
- Added in-memory caching to `wheelSizeApi.ts`
- Cache TTLs: Makes/Models/Years = 24h, Modifications = 12h, Vehicle data = 7 days
- Added cache utility functions (getCacheStats, pruneExpiredCache, clearCache)
- Commit: `ff81933` - "Add in-memory caching to Wheel-Size API to reduce API calls"

**Expected impact:** ~400 API calls/day max (vs 10,000) with normal usage patterns.

## Submodel/Trim Supplements for Older Vehicles

**Issue:** 2010 Chevy Avalanche (and other older vehicles) only showing "Base" trim instead of actual trims like LS, LT, LTZ. This happens when Wheel-Size API only returns engine codes (e.g., "5.3i") instead of trim names.

**Root Cause:**
- Wheel-Size API stores modifications by engine type, not trim level for older vehicles
- The "poor data" detection in trims endpoint wasn't catching labels like "Base (5.3i)"

**Fix deployed:**
1. Added submodel supplements for:
   - Chevrolet Avalanche (2002-2006: 1500, Z66, Z71, 2500; 2007-2013: LS, LT, LTZ)
   - Older Silverado/Silverado 1500 (1999-2006, 2007-2013 with proper trims)
   - Older Tahoe (2000-2006, 2007-2014)
   - Older Suburban (2000-2006, 2007-2014)

2. Fixed "poor data" detection in trims endpoint:
   - Now properly detects `"Base (5.3i)"` as poor data (was only checking exact `"base"`)
   - Added pattern to catch engine codes in parentheses

- Commit: `526e9a7` - "Add submodel supplements for Avalanche, older Silverado/Tahoe/Suburban + fix poor-data detection"

**Result:** Older GM trucks/SUVs now show proper trim options (LS, LT, LTZ, Z71, etc.) from static supplements.

## Restored Wheel-Size Catalog Data Layer (~1:20 PM)

**Issue:** Invalid vehicle combinations possible (e.g., 2006 Buick Encore which doesn't exist - Encore started 2013). The years dropdown was returning a static list (2000-2027) with no validation.

**Root cause:** We over-removed when trying to comply with ToS. Catalog endpoints ARE allowed to be cached.

**Solution - New catalog-store.ts:**
- Persistent JSON file storage at `data/catalog/wheel-size-catalog.json`
- Stores: makes, models (with valid years array), modifications
- Only caches catalog endpoints (NOT search API)
- Auto-populates from API on first access, then uses cache

**Updated Routes:**
1. **years route** - Now validates against catalog, returns only valid years for make/model
2. **models route** - Uses catalog, populates on API fetch
3. **trims route** - Validates year against catalog before API call (rejects invalid combos)

**New Admin Endpoint:**
- `POST /api/admin/catalog` with actions:
  - `populate-common` - Populate ~32 common US makes with models and years
  - `populate-models` - Populate models for a specific make
  - `populate-modifications` - Populate modifications for year/make/model
  - `clear` - Clear catalog

- Commit: `c43c147` - "Restore Wheel-Size catalog data layer with persistent storage"

**To initialize catalog:**
```bash
curl -X POST https://shop.warehousetiredirect.com/api/admin/catalog \
  -H "Content-Type: application/json" \
  -d '{"action": "populate-common"}'
```

This will take ~5 minutes and populate all common US makes with their models and valid years.
