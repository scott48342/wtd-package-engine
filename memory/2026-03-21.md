## 12:36 PM - Canonical modificationId Fix (trims route)

**Issue**: modificationId in trims selector response was NOT a true canonical fitment identity.

**Investigation findings**:
1. **Wheel-Size API** returns modifications with hex-based slugs (e.g., 5d0e19c6c5, e30b975915) - these ARE unique per modification
2. **Supplements** (submodel-supplements.json) were returning modificationId: "ltz" - just a slugified label, NOT unique
3. **Dedupe logic** collapsed multiple mods with same displayTrim, losing unique modificationIds

**Example of problem**:
```
// API returns:
{ slug: "5d0e19c6c5", name: "2.0T" }  // LS, LT, 1LS, 1LT, 1LE, 2LT, 3LT
{ slug: "e30b975915", name: "6.2i" }  // LT1, SS, SS 1LE, 1SS, 2SS

// Supplements returned:
{ value: "ltz", label: "LTZ", modificationId: "ltz" }  // ŌØī Not unique!
```

**Fix** (src/app/api/vehicles/trims/route.ts):

1. **Added makeSupplementId() function**:
   - Generates deterministic hash-based IDs for supplements
   - Format: s_{8-char-hash} where hash = SHA256(year:make:model:trimValue)
   - Now: { value: "s_1e42a3a7", label: "LS", modificationId: "s_1e42a3a7" }

2. **Fixed dedupe logic**:
   - Changed from dedupe-by-displayTrim to dedupe-by-modificationId
   - When multiple mods share same displayTrim but different IDs, disambiguate labels:
     - If rawTrim differs: "Base (5.7i)" vs "Base (3.8i)"
     - Otherwise: "Base #1", "Base #2"

**Result**:
```
// Before: { value: "ltz", modificationId: "ltz" }  ŌåÉ collision risk
// After:  { value: "s_ef9046fb", modificationId: "s_ef9046fb" }  ŌåÉ unique
```

**Files changed**: src/app/api/vehicles/trims/route.ts

## 9:30 PM - ModificationId Alias Mapping Implementation

### Design: Canonical Mapping with Alias Resolution

**Problem:**
- Trims API returns modificationId A (e.g., "98ad05ab24")
- Profile import gets different modificationId B from Wheel-Size API (e.g., "d23e2b5f89")
- DB lookup with A fails because data stored under B
- Result: HTTP 500 "Import succeeded but profile not found"

**Solution: modification_aliases table + alias resolution flow**

### New Schema: modification_aliases
\\\sql
CREATE TABLE modification_aliases (
  id UUID PRIMARY KEY,
  year INTEGER NOT NULL,
  make VARCHAR(100) NOT NULL,
  model VARCHAR(100) NOT NULL,
  requested_modification_id VARCHAR(255) NOT NULL,  -- what user requested
  canonical_modification_id VARCHAR(255) NOT NULL,  -- what's in vehicle_fitments
  display_trim VARCHAR(255),
  vehicle_fitment_id UUID REFERENCES vehicle_fitments(id),
  created_at TIMESTAMP DEFAULT NOW()
);
\\\

### New Resolution Flow

**Before (broke when A ? B):**
1. Lookup by A ? miss
2. Fallback to legacy ? import with different ID ? lookup by A ? miss
3. HTTP 500

**After (handles A ? B mapping):**
1. Lookup by A ? miss
2. Check modification_aliases for A ? found B
3. Lookup by B ? hit! Return profile
